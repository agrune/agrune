# Agrune Annotate Workflow

이 문서는 Agrune를 사용하기 위해 필요한 어노테이션 워크플로의 source of truth입니다.

하네스별 스킬, 플러그인, 에이전트 지침은 이 문서를 복제해서 제각각 진화시키지 말고, 여기 정의된 절차를 각 하네스 문법으로 얇게 감싸는 방식으로 유지합니다.

## 목표

- 사람이 실제로 조작할 인터랙션만 `data-agrune-*`로 노출한다
- 앱의 소스 코드 레벨에서 어노테이션한다
- DOM 구조가 바뀌어도 의미가 유지되도록 이름과 그룹을 설계한다
- 민감 입력은 `data-agrune-sensitive`로 표시한다

## 기본 절차

1. 앱 구조를 파악한다.
2. 사용자가 실제로 클릭, 입력, 드래그, 대기할 요소를 찾는다.
3. 가능한 한 소스 컴포넌트 레벨에서 어노테이션을 추가한다.
4. 관련 요소는 `data-agrune-group`으로 묶어 스냅샷이 읽기 쉽게 만든다.
5. 액션 가능한 요소에는 `data-agrune-action`과 `data-agrune-name`을 붙인다.
6. 비밀번호, 토큰, 결제 정보 입력처럼 민감한 값은 `data-agrune-sensitive`를 붙인다.
7. 변경 후 앱을 다시 스캔해 중복되거나 애매한 이름이 없는지 확인한다.

## 네이밍 원칙

- 사용자 관점에서 바로 이해되는 이름을 쓴다
- "button1", "input-field" 같은 구현 중심 이름은 피한다
- 같은 화면 안에서 같은 이름이 반복되면 그룹으로 구분한다
- 가능한 한 UI에 보이는 문구와 의미를 맞춘다

## 그룹 원칙

- 폼, 모달, 사이드바, 툴바, 캔버스 패널처럼 의미 단위가 있는 경우 그룹을 만든다
- 그룹 이름은 화면 구조가 아니라 사용자 작업 단위에 맞춘다
- 지나치게 잘게 쪼개지 않는다

## 피해야 할 것

- 실제 인터랙션이 불가능한 장식 요소를 어노테이션하는 것
- 빌드 산출물이나 임시 HTML을 직접 수정하는 것
- 런타임에 잠깐 생기는 내부용 DOM을 타깃으로 잡는 것
- 민감 필드를 일반 입력 필드처럼 노출하는 것

## 결과물 기대치

- 어노테이션된 소스 코드 변경
- 새로 추가한 그룹/타깃의 의미가 드러나는 간단한 설명
- 필요하면 스냅샷에서 어떻게 보일지 예시
